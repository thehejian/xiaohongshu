## 📖 文章正文

用 AI 写代码一时爽 🥳
**一直用一直爽?不,火葬场。😅**

接口参数靠猜,运行报错了才傻眼;
让它加个功能,它把整个架构改崩了;
问你能不能做,它说"可以可以",结果全是幻觉……

**这不是 AI 的错,是它没规矩。**

Vibe Coding(氛围编程) 时代,有人总结了 **AI 编程八荣八耻**,让 AI 学会"做人"👇

---

### 🏆 AI 编程八荣八耻

| 荣(要这样做) | 耻(别这样干) |
|------|------|
| 认真查阅文档 | 暗猜接口参数 |
| 寻求确认再执行 | 模糊执行不管后果 |
| 找人类确认业务 | 自己想象需求 |
| 复用现有代码 | 创造新接口/轮子 |
| 主动测试验证 | 跳过验证直接交差 |
| 遵循代码规范 | 破坏现有架构 |
| 诚实说"不知道" | 假装理解硬撑 |
| 谨慎重构 | 盲目修改 |

---

### 逐条解析 ✨

**1️⃣ 查文档 vs 猜参数**
- ❌ "这个 API 应该是这样传参吧"
- ✅ "让我查一下官方文档确认"

**2️⃣ 确认 vs 莽撞**
- ❌ "用户说要个登录,直接开干"
- ✅ "先确认:密码登录?验证码?第三方?"

**3️⃣ 业务确认 vs 自己编**
- ❌ "我觉得这个需求应该是这样"
- ✅ "这个业务逻辑需要你确认一下"

**4️⃣ 复用 vs 造轮子**
- ❌ "这个功能没现成的,自己写一个"
- ✅ "先看看项目里有没有类似实现"

**5️⃣ 测试 vs 跳过**
- ❌ "代码写完就行,测什么测"
- ✅ "跑一下测试用例,确保没问题"

**6️⃣ 规范 vs 放飞**
- ❌ "这个逻辑放哪里都行,随便插"
- ✅ "按项目目录结构来,别破坏架构"

**7️⃣ 诚实 vs 硬撑**
- ❌ "我理解了,这样那样就行"
- ✅ "这个地方我不确定,需要你解释一下"

**8️⃣ 谨慎 vs 乱改**
- ❌ "这段代码有点丑,优化一下"
- ✅ "如果要改,先确保不影响现有功能"

---

### 💡 为什么这八条重要?

Vibe Coding 追求快,但快 ≠ 糙。

AI 最大问题:**幻觉 + 过度自信**
- 它觉得自己懂,其实在猜
- 它觉得自己能,其实在编

**这八荣八耻,就是给 AI 立的"规矩":**
👉 不确定 → 问
👉 不明白 → 说
👉 要动手 → 先验
👉 改代码 → 小心

把它放进你的 System Prompt 或 CLAUDE.md,代码质量立竿见影提升 💪

---

### 📦 怎么用?

**直接喂给 AI:**
> "请遵循以下编程原则:以暗猜接口为耻,以认真查阅为荣……"

**或者设为系统提示词,让它天生自带"规矩"。**

> **Vibe Coding 时代,AI 不是不懂,是没规矩。**
> **八荣八耻,治好人机协作的"默契"。**
